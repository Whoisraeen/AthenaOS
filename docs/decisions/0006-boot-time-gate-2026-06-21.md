# ADR 0006 — Boot-time gate: death-by-smoketests, deferral plan (2026-06-21)

- Status: accepted (plan; routed to next kernel slot)
- Date: 2026-06-21
- Owner: raeen-lead (autonomous), measured by raeen-perf

## Context
The one machine-enforced gate AthenaOS misses: `[boot] WARN: boot time 8500ms exceeds
6000ms target!` (QEMU TCG -smp 2, HEAD 544a696; ~11-13s on iron). raeen-perf measured a
repeatable per-tier breakdown (TSC-ms, sub-step `*-prof` instrumentation already in-tree):

| Phase | Δms | dominant |
|---|---|---|
| Tier 1 core | 2096 | modules=1740 |
| Tier 6 USB | 924 | xhci_smoke=858 |
| Tier 7 power | 1100 | aer+modules=1098 (iron=ACPI 159-dev parse; QEMU trivial) |
| Tier 8 platform | 1603 | audio/gpu/rgb/theme/vibe/raebridge/bt/edid smoketests |
| post-Tier-9 | ~1413 | vnet_probe=983, raefs=402 |

Diagnosis: **no single 4s hog — boot is dozens of synchronous correctness smoketests run
serially on CPU0 before the success marker.** The win is deferring whole CLASSES of
non-critical smoketest off the boot critical path, not micro-tuning. Deferring also helps
iron MORE than QEMU because each deferred test removes its `serial_println!` lines (the
~5ms/line UART + GOP-blit iron logging tax, memory `iron-console-logging-tax`).

## Decision
Three ranked, low-risk deferrals + one honesty fix. All are in `kernel/src/` (ONE crate →
serialized; they take the kernel slot AFTER the compositor-shadow UI work, ADR 0005):

1. **vnet_probe (983ms) → defer.** The boot-path virtio-net RX bring-up probe is a once-proven
   diagnostic; the real DHCP bind already runs in `net::spawn_poll_thread()`. Move the probe
   into that post-boot poller or gate behind `RAEEN_NET_PROBE`. Owner: raeen-drivers
   (`net_drivers.rs` + main.rs call site → serialize).
2. **xhci_smoke (858ms) → defer the empty-port walk.** Keep the cheap controller register-check
   + HID arming on the critical path; move the slow per-port GetPortStatus/transfer-wait scan
   into the HID input thread / a one-shot post-boot worker. MUST preserve `spawn_hid_input_thread`
   arming behavior (memory `hid-drain-edf-latency`). Owner: raeen-drivers (`xhci.rs`,
   self-contained — no main.rs edit).
3. **Tier1+Tier8 userspace-feature smoketests (~3.3s) → post-boot self-test sweep.** Batch the
   clearly-non-critical set (theme/vibe/wallpaper/rgb/widgets/search/game_profile/wireguard/
   bluetooth) into a `boot_selftest::run_deferred()` that runs AFTER the success marker. They
   still print PASS/FAIL (R10 rule 16 preserved) — just not gating the marker. Owner:
   raeen-kernel (`main.rs` → serialize; highest coordination).
4. **BOOT-BENCH honesty fix (never-fake-green).** `[BOOT-BENCH] T0->userspace` reads OK (4088ms)
   while the real gate fails (8500ms) because it snaps its end BEFORE the post-boot spawns +
   net/raefs post-9 work — it masks the red gate. Move its end-snap to the same point as
   `record_boot_complete()` (fast_boot.rs:1501) so there is ONE honest number. Owner:
   raeen-kernel (`fast_boot.rs`/`main.rs`).

## Rationale
Tie-breaker: (3) simplest correct — deferral changes no behavior, only ordering; (4) board
honesty is non-negotiable (charter: never fake green). None touch input->present, audio, or
SCHED_BODY steady-state paths. Iron-only Tier-7 ACPI cost is NOT QEMU-tunable → pending iron.

## How to reverse
Each deferral is independent; re-inline any smoketest by moving its call back ahead of the
marker. The BOOT-BENCH snap point is a one-line move.
