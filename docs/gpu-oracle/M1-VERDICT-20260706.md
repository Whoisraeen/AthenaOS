# M1 VERDICT — real-amdgpu iron run (2026-07-06)

**Campaign:** GPU real-submit (`docs/superpowers/specs/2026-07-06-gpu-real-submit-design.md`), milestone M1.
**Question M1 asked:** does the REAL upstream amdgpu C (Strategy B) clear the MES wall under our LinuxKPI shim on Athena?
**Answer:** The real init runs FAR past the native-Rust MES halt — it never reaches MES this run; it wedges much earlier, at **IP-discovery register-base enumeration of the CLKA (clock, v13.0.3) block, instance #2.** This is a NEW, deeper wall than the native path's `0x7654`, and it is almost certainly a shim/loader bug, not a GPU-hardware limit.

## How we got the answer (the diagnostic that worked)

The first M1 boot (earlier 2026-07-06) wedged with ZERO capture + no auto-return (manual power cycle). Root cause: the real `amdgpu_device_init` ran at Game scheduling priority (promoted in `_start`, never demoted), so a CPU-0 wedge starved the safe-progress broadcaster + auto-return threads. Fix (`62ffb18`, `[interface]`):
- **`SYS_NETLOG_FLUSH` (296)** — synchronous, on-demand UDP broadcast of the bootlog ring (safe-mode-safe, CPU-0-inline, thread-independent).
- **`flush_marker()` fences** at sentinels 9000 (daemon start) / 9001 (probe+discovery) / 9002 (entering the C init) / 9003 (C init returned).
- Demote to Normal before the C call (attempt self-recovery).
- Fixed `netlog-listen.ps1` `Save-Log` absolute-path crash that lost the first boot's capture.

Second boot captured **122,869 bytes** of netlog (`netlog-M1-REAL-AMDGPU-20260706.txt`; readable ladder in `M1-amdgpu-trail-20260706.txt`).

## The trail (what the real init achieved on iron)

- `msg: 9000` — amdgpud `_start` ran (rules OUT a pre-userspace / scheduler wedge)
- `[amdgpu] firmware preflight: loader=ok amdgpu_blobs=12/12 present` — all Phoenix blobs loaded
- `CKPT 0` → `DISC 1..5: parsed 39 IP blocks — SOC15 offsets ACTIVE` — IP discovery blob parsed, first-dword `0x28211407` matched
- `probe-time SMU mailbox check: GetSmuVersion->Some(1)` — SMU responsive
- `msg: 9001` — `CKPT 1: probe OK + DISCOVERY ACTIVE`
- `msg: 9002` — **`REAL-INIT: entering amdgpu_device_init (C)`** — reached the real upstream init
- `RELOC-CHK nbio_v4_3_funcs@0x17aea0 slot35(set_reg_remap)=0x146a46 (want ~0x12bc2c) slot0(get_rev)=0x0 (want ~0x12b96f)` — the `-fvisibility=hidden`/`-Bsymbolic` fix (`c0e350d`) relocated slot35 but **slot0 is still NULL** (a partial-reloc anomaly worth a look, though init continued past it)
- `initializing kernel modesetting (IP DISCOVERY 0x1002:0x15BF … 0xC3)` → `number of hardware IPs on die0: 39`
- `set register base offset for ATHUB` → `CLKA(6) #0 v13.0.3:` (+base values) → `CLKA #1:` (+base values) → **`set register base offset for CLKA` → `CLKA(6) #2 v13.0.3:` … FREEZE** (no base-address values printed for #2; **no `msg: 9003`**)
- `System successfully booted` IS present — the OS is healthy; only the GPU daemon's C init wedged.

## Verdict: the wall moved from MES `0x7654` to IP-discovery CLKA #2 enumeration

This is **not** branch A/C/D of the plan's table cleanly — the real init got much further than expected and died at a spot the native path never reached. Classified as a **shim/loader bug (plan branch B family), pre-MES:**

**Primary hypothesis (highest odds):** the freeze is in the **linuxkpi C vararg-print facade**, not hardware. The `CLKA #0/#1` lines printed their `0x…` base-address VALUES; `#2` froze *after* the header, *before* its value — i.e. inside the print of the base address. The 2026-07-04 WIP added `rae_dbg_ptrs` precisely because "the C `vscnprintf`-varargs path faults on `%lx`" — the `dev_info`/`DRM_DEBUG` of instance #2's base is a strong candidate for that fault path. If so, the fix is in `raeen_linuxkpi`'s printf facade (or bypassing/So the discovery enumeration doesn't vararg-print), and the init would continue.

**Secondary hypothesis:** an out-of-bounds harvest-table read for the 3rd CLK instance in the discovery walk (`amdgpu_discovery.c`), i.e. a real parse bug for this Phoenix die's CLK instancing.

**Ruled out this run:** the MES `0x7654` halt (never reached), the KVM near-null-write scheduler bug (OS booted healthy, 9000/9001/9002 all fired), a pre-userspace hang.

## Next step (M1.1, before any M2 work)

1. Localize CLKA-#2: add a `flush_marker` / netlog fence INSIDE the discovery reg-base loop (per-instance), and/or route that print away from the C `%lx` vararg path (use the `rae_dbg_ptrs`-style Rust klog). One more iron boot then says "printf-facade fault" vs "discovery parse OOB."
2. If printf-facade: harden `raeen_linuxkpi` `vscnprintf`/`%lx`. If parse OOB: fix the CLK-instance harvest walk.
3. Investigate the `slot0=0x0` nbio vtable partial-reloc as a parallel item (may bite a later IP block even after CLKA is fixed).

The self-recovery half is still open: demote-to-Normal did NOT recover the box (an MMIO/print hard-hang on CPU 0 is uninterruptible). Manual power cycle still required per wedge. Widening that is secondary to localizing CLKA-#2.

## Bottom line

**M1 is a success as a diagnostic milestone:** the real amdgpu driver provably runs on Athena, loads all firmware, parses IP discovery, brings SMU up, and executes deep into `amdgpu_device_init` — dying at CLKA-#2 register-base enumeration, a specific, shim-level, fixable wall well before MES. The Strategy-B path is viable; the next lever is small and named.
